Intellacc E2EE Architecture

This page captures the active browser E2EE design in the Intellacc repo: a split-key local vault, OpenMLS WASM for group encryption, and a backend relay that stores ciphertext and delivery metadata.

Generated from repository inspection on 2026-05-22.

One-Screen Summary

Vault Layer

Protects MLS state, decrypted message cache, TOFU fingerprints, and processed-message IDs in IndexedDB. The vault key is derived from both the server-gated Master Key and the device-local Local Key.

MLS Layer

OpenMLS compiled to WASM creates identities, key packages, groups, welcomes, commits, proposals, application ciphertexts, AAD, and confirmation tags.

Relay Layer

Express/PostgreSQL routes key packages, welcomes, commits, and application ciphertexts to recipient devices. It should not receive plaintext message bodies or local MLS ratchet state.

Key Hierarchy

The core vault design is split-key. An offline server database dump alone should not decrypt local MLS state because the Local Key is device-local. A disk or browser-profile copy alone should not decrypt it because the Master Key is server-gated and wrapped. A server that can ship malicious browser code is a different threat model and can attack unlocked clients. 

flowchart TD
  P[User password] --> A[Argon2id wrapping key]
  PRF[WebAuthn PRF output] --> H[HKDF PRF wrapping key]

  A --> MKW[Server: password-wrapped Master Key]
  H --> MKP[Server: PRF-wrapped Master Key]

  A --> LKW[IndexedDB: password-wrapped Local Key]
  H --> LKP[IndexedDB: PRF-wrapped Local Key]

  MK[Master Key]
  LK[Local Key]
  MK --> C[Composite Key = SHA-256 of Master Key plus Local Key]
  LK --> C

  C --> V[Encrypted IndexedDB vault]
  V --> MLS[MLS identity and group ratchet state]
  V --> MSG[Encrypted local plaintext cache]
  V --> FP[Encrypted contact fingerprints]
  V --> DEDUPE[Processed message IDs]
Item Where It Lives What It Protects
Master Key Wrapped on backend in user_master_keys One half of the composite vault key. Server-gated by trusted device checks.
Local Key Wrapped in browser IndexedDB Device-local half of the composite vault key.
Composite Key Memory only while unlocked Encrypts local MLS state, message cache, fingerprints, and dedupe records.
MLS State Encrypted in IndexedDB granular storage OpenMLS identity, groups, epochs, secrets, pending commits, and ratchets.

Unlock Path

Unlocking is not just password login. The browser must also present a trusted local device id so the backend will release the wrapped Master Key. The device must have been verified after the Master Key was last updated. 

sequenceDiagram
  participant Browser
  participant Backend
  participant IndexedDB
  participant OpenMLS

  Browser->>Backend: Login and receive JWT
  Browser->>IndexedDB: Read local device ids
  Browser->>Backend: GET /users/master-key with x-device-ids
  Backend->>Backend: Check user_devices.last_verified_at >= user_master_keys.updated_at
  Backend-->>Browser: Wrapped Master Key or LINK_REQUIRED
  Browser->>Browser: Unwrap Master Key with password or PRF
  Browser->>IndexedDB: Load wrapped Local Key
  Browser->>Browser: Unwrap Local Key
  Browser->>Browser: Derive Composite Key
  Browser->>IndexedDB: Decrypt MLS state
  Browser->>OpenMLS: Restore identity, groups, ratchets
  Browser->>Backend: Refresh/upload MLS key packages

First Device And Device Linking

The first device is implicitly trusted because it creates the first Master Key and local vault. Later devices must be approved by an already verified device, with the approver password and x-device-id. 

sequenceDiagram
  participant NewDevice
  participant Backend
  participant TrustedDevice

  NewDevice->>Backend: POST /devices/link/start with new device_public_id
  Backend-->>NewDevice: Short-lived linking token
  Backend-->>TrustedDevice: deviceLinkRequest event
  TrustedDevice->>Backend: Approve token + approver_password + x-device-id
  Backend->>Backend: Verify password
  Backend->>Backend: Verify approving device is trusted
  Backend->>Backend: Insert/update new user_devices row with last_verified_at
  NewDevice->>Backend: Poll link status
  NewDevice->>Backend: GET /users/master-key with local device id
  Backend-->>NewDevice: Wrapped Master Key

MLS Bootstrap And Key Packages

Once unlocked, the client creates or restores an OpenMLS identity. It keeps a pool of public key packages on the backend so other users can add each device to MLS groups while that device is offline. 

flowchart LR
  U[Unlocked browser device] --> W[OpenMLS WASM client]
  W --> I[Identity: user id as MLS credential]
  W --> K[Generate key packages]
  K --> R[Regular package pool]
  K --> L[Last-resort package]
  R --> B[Backend mls_key_packages]
  L --> B
  B --> O[Other clients fetch packages for invitations]

Regular key packages are consumed when fetched. Last-resort packages are retained as fallback. The client also checks package lifetime ranges before accepting packages.

Starting A Direct Message

A direct message is still an MLS group. The backend tracks the pair in mls_direct_messages, while the clients perform the actual MLS group creation and member addition. 

sequenceDiagram
  participant Alice
  participant Backend
  participant Bob

  Alice->>Backend: POST /mls/direct-messages/:bobId
  Backend->>Backend: Find or create deterministic DM group id
  Backend-->>Alice: groupId and isNew

  alt New DM
    Alice->>Alice: Create local MLS group
    Alice->>Backend: Fetch Bob key packages, all devices
    Alice->>Alice: OpenMLS add_member per Bob key package
    Alice->>Backend: Relay commit to existing members only
    Alice->>Backend: Relay welcome to Bob devices
  else Existing DM
    Alice->>Alice: Require local group state
    Alice->>Backend: Optionally sync pending welcomes
  end

  Bob->>Backend: GET pending queue
  Bob->>Bob: Stage welcome
  Bob->>Bob: Validate members and sender
  Bob->>Bob: Accept or keep pending
  Bob->>Backend: Ack welcome after acceptance

Welcome Handling

Welcomes are staged before acceptance. The client validates credentials, lifetimes, and DM membership rules before joining. Contacts followed by the receiver may be auto-accepted. Other welcomes can remain pending for UI approval. 

flowchart TD
  Q[Pending welcome from relay queue] --> S[stage_welcome in WASM]
  S --> V{Validation passes?}
  V -- No --> R[Reject staged welcome and mark processed]
  V -- Yes --> F{Receiver follows sender?}
  F -- Yes --> A[Accept staged welcome]
  F -- No --> P[Keep pending or ask UI handler]
  P --> D{User/handler decision}
  D -- Accept --> A
  D -- Reject --> R
  A --> M[Persist MLS state]
  A --> KP[Regenerate key package]
  A --> G[Sync group members to backend]
  A --> ACK[Ack welcome]

Sending And Receiving Messages

Message plaintext is encrypted by OpenMLS in the browser. The backend stores and forwards the MLS ciphertext. The receiver decrypts locally and then stores the plaintext only inside the encrypted local vault. 

sequenceDiagram
  participant Sender
  participant MLS1 as Sender OpenMLS
  participant Backend
  participant MLS2 as Receiver OpenMLS
  participant Receiver

  Sender->>MLS1: encrypt_message(groupId, plaintext)
  MLS1-->>Sender: MLS ciphertext with AAD
  Sender->>Backend: POST /mls/messages/group with application ciphertext
  Backend->>Backend: Verify sender device and group membership
  Backend->>Backend: Store in mls_relay_queue
  Backend->>Receiver: Socket event / push notification
  Receiver->>Backend: GET /mls/queue/pending
  Receiver->>MLS2: decrypt_message_with_aad(groupId, ciphertext)
  Receiver->>Receiver: Validate AAD groupId, epoch, type
  Receiver->>Receiver: Persist plaintext encrypted in local vault
  Receiver->>Backend: POST /mls/queue/ack

Commit And Membership Flow

Membership changes produce commits. The sender only merges a pending commit after the relay accepts it. The backend enforces commit epoch uniqueness for a group, and fans commits to existing members. 

flowchart TD
  A[Client wants membership/key update] --> B[Create rollback state]
  B --> C[Set AAD type = commit]
  C --> D[OpenMLS creates commit]
  D --> E[POST commit to relay with groupId and epoch]
  E --> F{Relay accepts?}
  F -- No --> G[Clear pending commit and restore rollback]
  F -- Yes --> H[merge_pending_commit]
  H --> I[Persist new MLS state]
  I --> J[Broadcast confirmation tag]
  J --> K[Other clients compare tags for fork detection]

Backend Relay Data Model

The relay has enough information to route messages, not enough to read them. It does know social and delivery metadata.

What The Server Can And Cannot See

Visible To Backend Not Visible To Backend In Normal Operation
User ids, device ids, group ids, DM pair metadata Plaintext message bodies
Message type: welcome, commit, proposal, application Local MLS ratchet state and group secrets
Ciphertext blobs and delivery/ack state Decrypted local message cache
Key packages and package lifetime metadata Contact safety-number verification state contents
Generic push-notification event for application messages Composite vault key, Local Key, unwrapped Master Key

AAD, Dedupe, And Fork Detection

AAD Binding

Application, commit, and proposal messages bind groupId, epoch, and type. If decrypted AAD does not match the expected context, the client marks the message invalid.

Persistent Dedupe

Processed relay queue ids are stored in IndexedDB so already handled messages are skipped across page reloads and restarts.

Confirmation Tags

After commits, clients send encrypted system messages containing confirmation tags. Mismatches indicate a possible group fork. 

flowchart TD
  D[Decrypt MLS application message] --> A{AAD matches groupId, epoch, type?}
  A -- No --> X[Mark invalid and skip UI]
  A -- Yes --> P{Plaintext is confirmation tag?}
  P -- Yes --> T[Compare local and remote confirmation tags]
  T --> F{Mismatch?}
  F -- Yes --> W[Emit fork warning]
  F -- No --> S[Skip as system message]
  P -- No --> M[Persist plaintext encrypted locally]
  M --> U[Display in message UI]

Safety Numbers And TOFU

Contact fingerprints are based on MLS signature keys. The first seen fingerprint is stored in the encrypted vault. If the fingerprint changes, verification is reset and the UI can warn about a possible MITM or device reset. Users can compare safety numbers out of band. 

flowchart LR
  W[Welcome/member info] --> S[Extract signature key]
  S --> F[Derive fingerprint]
  F --> E{Existing fingerprint?}
  E -- No --> N[Save as first seen, unverified]
  E -- Same --> O[Keep current status]
  E -- Changed --> C[Store previous fingerprint, mark changed]
  N --> V[User may verify safety number]
  O --> V
  C --> ALERT[Show warning]
  V --> DONE[Mark verified in encrypted vault]

Password And Passkey PRF Unlock

Password unlock and WebAuthn PRF unlock are dual wrapping paths. Both paths must recover the same Master Key and Local Key before the composite vault key can be derived. 

flowchart TD
  subgraph Password path
    P[Password] --> PA[Argon2id]
    PA --> PMK[Unwrap server Master Key]
    PA --> PLK[Unwrap local Local Key]
  end

  subgraph Passkey PRF path
    PRF[Authenticator PRF output] --> HK1[HKDF MasterKey_PRF]
    PRF --> HK2[HKDF LocalKey_PRF]
    HK1 --> RMK[Unwrap server PRF Master Key]
    HK2 --> RLK[Unwrap local PRF Local Key]
  end

  PMK --> MK[Master Key]
  RMK --> MK
  PLK --> LK[Local Key]
  RLK --> LK
  MK --> C[Composite vault key]
  LK --> C

Main Caveat

This is browser-based E2EE. The backend database should not be able to decrypt stored MLS ciphertexts, but the web client still trusts the server to deliver honest JavaScript and WASM. A malicious or compromised deployment pipeline could serve altered client code. That is the normal caveat for web-delivered E2EE.
